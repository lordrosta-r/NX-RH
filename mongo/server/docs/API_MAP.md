## Auth & Sécurité

**Mécanisme** : Cookie httpOnly (`token`) contenant un JWT signé HS256.

**Middleware** : `authGuard(roles?)` — valide le JWT, vérifie `isActive` en DB, attache `req.user = { id, email, role }`.
- Appelé sans argument (`authGuard()`) → tout utilisateur authentifié.
- Appelé avec une liste → seuls les rôles listés sont autorisés (403 sinon).

**Routes publiques** : `GET /api/health`, `POST /api/auth/login`, `POST /api/auth/logout`

**Rate limiting global** :
- `apiLimiter` : 2 000 req/min sur tout `/api/`
- `mutationLimiter` : 500 req/min sur les routes d'écriture (users, campaigns, forms, events, resources, offboarding, evaluations/bulk)

**Rôles valides** : `admin`, `hr`, `manager`, `employee` (+ `director` en legacy — voir §Inconsistances)

**Montages** (depuis `index.js`) :

| Préfixe montage | Middleware de montage |
|---|---|
| `/api/auth` | aucun |
| `/api/users` | `mutationLimiter`, `authenticated` |
| `/api/campaigns` | `mutationLimiter`, `authenticated` |
| `/api/forms` | `mutationLimiter`, `authenticated` |
| `/api/evaluations/bulk` | `mutationLimiter` *(sans `authenticated` — voir §Anomalies)* |
| `/api/evaluations` | `authenticated` |
| `/api/analytics` | `apiLimiter`, `authenticated` |
| `/api/events` | `mutationLimiter`, `authenticated` |
| `/api/resources` | `mutationLimiter`, `authenticated` |
| `/api/admin/ldap` | `mutationLimiter`, `authGuard(['admin'])` |
| `/api/admin/audit` | `apiLimiter`, `authGuard(['admin', 'hr'])` |
| `/api/admin` | `mutationLimiter`, `authGuard(['admin'])` |
| `/api/offboarding` | `mutationLimiter`, `authenticated` |

`authenticated` = `authGuard(['admin', 'director', 'manager', 'employee', 'hr'])`

| Method | Path | Auth | Roles | Query | Body | Response | Codes |
|--------|------|------|-------|-------|------|----------|-------|
| POST | `/api/auth/login` | Non | — | — | `{ email*, password*, remember? }` | `{ user: { id, email, firstName, lastName, role } }` | 200, 400, 401, 429 |
| POST | `/api/auth/logout` | Non | — | — | — | `{ message }` | 200 |
| GET | `/api/auth/me` | Oui | tous | — | — | `{ id, _id, firstName, lastName, email, role, department, position, isActive, locale, theme, notificationPrefs, lastLoginAt, authSource, managerId, onboarding, createdAt }` | 200, 401 |

**Détails POST /login** :
- `remember` (bool, optionnel) : étend le cookie à 30j (défaut : 8h / `JWT_EXPIRES_IN`)
- Double rate limiter : 5 tentatives/15min par email + 20/15min par IP
- Met à jour `lastLoginAt` en fire-and-forget

**Détails GET /me** :
- Efface le cookie et retourne 401 si le compte est inactif
- `notificationPrefs` est filtré selon le rôle (seules les clés autorisées sont retournées)
- ⚠️ Retourne à la fois `id` (string) et `_id` (ObjectId) avec la même valeur

**Détails PATCH /preferences** :
- `locale` : valeurs autorisées depuis `config/constants.LOCALES`
- `theme` : valeurs autorisées depuis `config/constants.THEMES`
- `notificationPrefs` : merge avec l'existant (pas d'écrasement total) ; clés inconnues → 400 ; clés hors-rôle → 403 ; valeurs non-bool → 400

| Method | Path | Auth | Roles | Query | Body | Response | Codes |
|--------|------|------|-------|-------|------|----------|-------|

| GET | `/api/users/:id` | Oui | tous | — | — | User object | 200, 400, 403, 404 |
| POST | `/api/users` | Oui | admin, hr | — | `{ firstName*, lastName*, email*, role?, department?, position?, managerId? }` | `{ ...user, tempPassword }` | 201, 400, 403, 409 |
| PATCH | `/api/users/:id` | Oui | admin/hr ou self | — | `{ email?, firstName?, lastName?, department?, position?, role?, managerId?, isActive? }` | User object | 200, 400, 403, 404 |
| GET | `/api/users/:id/offboard-preview` | Oui | admin, hr | — | — | `{ user, pendingEvaluations, activeCampaigns }` | 200, 400, 403, 404 |
| PATCH | `/api/users/:id/offboard` | Oui | admin, hr | — | `{ reason*, effectiveDate* }` | User object | 200, 400, 403, 404 |
| PATCH | `/api/users/:id/onboarding/complete` | Oui | admin, hr ou self | — | — | User object | 200, 400, 403, 404 |
| PATCH | `/api/users/:id/onboarding/:stepIndex` | Oui | admin, hr ou self | — | `{ done? }` | User object | 200, 400, 403, 404 |
| GET | `/api/users/:id/gdpr-export` | Oui | admin, hr ou self | — | — | `{ user, evaluations, exportedAt }` *(Content-Disposition: attachment)* | 200, 400, 403, 404 |
| DELETE | `/api/users/:id/gdpr-anonymize` | Oui | admin uniquement | — | — | `{ success, anonymizedId }` | 200, 400, 403, 404, 409 |

**Détails GET /api/users** :
- `manager` : voit uniquement ses subordonnés directs (`managerId == req.user.id`)
- `employee` : bloqué (403) — ne peut pas lister les utilisateurs
- `isActive` : `'true'` / `'false'` (string) pour filtrer les actifs/inactifs ; sans ce paramètre, pas de filtre sur isActive
- `search` : regex insensible à la casse sur `firstName`, `lastName`, `email` (échappé contre ReDoS, max 100 chars)
- Sans `?page` : retourne un array JSON brut limité à 100. Avec `?page` : retourne `{ data, total, page, limit }`.
- `limit` max : 100

**Détails PATCH /api/users/:id** :
- Whitelist : `['email', 'firstName', 'lastName', 'department', 'position', 'role', 'managerId', 'isActive']`
- Non-admin : champs protégés = `['role', 'managerId', 'isActive', 'department', 'position', 'email']` → 403 si tentative de modification
- Champs exclus des réponses : `passwordHash`, `ldapDn`

**Détails POST /api/users** :
- ⚠️ `tempPassword` est stocké directement comme `passwordHash` sans bcrypt (raw hex string)
- `tempPassword` est exposé une seule fois dans la réponse 201

**Détails PATCH /offboard** :
- Passe `user.offboardingStatus = 'offboarding'`
- Archive toutes les évaluations en cours (`status` ≠ validated/archived) de cet utilisateur
- Fire-and-forget audit log

**Détails DELETE /gdpr-anonymize** :
- Bloqué si des évaluations avec statuts `['assigned', 'in_progress', 'submitted']` existent → 409
- Anonymise : firstName/lastName → 'Anonyme', email → `anonyme-{id}@deleted.local`, phone/avatar → null, isActive → false

| PATCH | `/api/evaluations/:id` | Oui | tous (scope par rôle) | — | `{ answers?, score?, reviewerComment?, nextObjectives?, objectiveRatings?, evaluateeComment?, status? }` | `{ id, status, lastSavedAt, reviewerComment, evaluateeComment }` | 200, 400, 403, 404, 409 |
| POST | `/api/evaluations/:id/expire` | Oui | admin, hr | — | — | `{ id, status }` | 200, 400, 403, 404, 409 |
| GET | `/api/evaluations/:id/pdf` | Oui | admin, hr ou participant | — | — | PDF attachment | 200, 400, 403, 404 |

**Détails GET /history** :
- Retourne les évaluations où l'utilisateur courant est évaluateur OU évalué, avec statuts terminaux/avancés
- Statuts inclus : `['submitted', 'reviewed', 'signed_evaluatee', 'signed_manager', 'signed_hr', 'validated']`
- Retourne `{ data: [...] }` (tableau dans un objet) — **sans** `total`, `page`, `limit`. Limité à 200 résultats.
- Peuple : `formId`, `evaluatorId`, `evaluateeId`, `campaignId`

**Détails GET /** :
- `employee` : ses propres évaluations uniquement (évaluateur ou évalué)
- `manager` / `director` *(legacy)* : ses évaluations + celles des subordonnés visibles (scope via `getVisibleUserIds`)
- `admin` / `hr` : tout
- Sans `?page` : array brut limité à 200 (marqué "Legacy" dans le code). Avec `?page` : `{ data, total, page, limit }`
- `status` : whitelist stricte via `EVALUATION_STATUSES`

**Détails GET /export** :
- Filtre `dept` appliqué **côté JS** après fetch (pas en MongoDB) — perf dégradée pour gros datasets
- Limite 5000 enregistrements
- Colonnes CSV : Évalué, Manager, Campagne, Statut, Score, Département, Date

**Détails POST /bulk** :
- Max 500 évaluations par batch
- `ordered: false` sur `insertMany` : continue malgré les doublons
- Retourne 201 `{ created: N }` si tout ok, ou **207** `{ created, skipped, message }` si des doublons ont été ignorés
- Gèle les forms concernés (fire-and-forget `frozenAt`)
- Notifie les évalués assignés (fire-and-forget)

**Détails PATCH /bulk** :
- `action` : `'archive'`, `'sign_hr'`, `'assign_reviewer'`
- `reviewerId` requis seulement pour `assign_reviewer`
- Max 200 IDs par opération
- ⚠️ L'action `'archive'` ne passe pas nécessairement au statut `'archived'` — elle prend le premier statut disponible dans les transitions valides pour l'évaluation

**Détails PATCH /:id** :
- `answers` : array `[{ questionId, value }]` — bloqué si statut dans `LOCKED_STATUSES`
- `score` : manager/director/admin/hr uniquement ; le manager doit être le managerId direct de l'évalué
- `reviewerComment`, `nextObjectives`, `objectiveRatings` : manager/director/admin/hr uniquement (max 5000 chars)
- `evaluateeComment` : évalué lui-même ou admin/hr (max 5000 chars)
- `status` : transitions contrôlées par `VALID_TRANSITIONS` (admin) ou `ROLE_TRANSITIONS[role]` (autres)
- Notifie automatiquement les parties prenantes selon le nouveau statut (fire-and-forget)
- Retourne **uniquement** `{ id, status, lastSavedAt, reviewerComment, evaluateeComment }` (pas l'évaluation complète)

**Détails PATCH /:id/reassign** :
- `newEvaluatorId` doit appartenir à un user actif avec rôle `'manager'` ou `'director'`
- Bloqué pour les statuts terminaux : `'signed_hr'`, `'validated'`
- L'ancienne affectation est tracée dans `evaluation.auditLog[]`

- `filename` validé par regex `^[a-zA-Z0-9_\-.]+$` + refus des `..`
- `status` et `visibleTo` non validés explicitement — délégués au schéma Mongoose

## Inconsistances observées

---

### 1. Format d'erreur non uniforme — Messages 403 incohérents

Les messages d'erreur 403 utilisent **3 conventions différentes** selon le fichier source :

| Fichier | Message 403 utilisé |
|---|---|
| `users.js`, `offboarding.js` | `'Insufficient permissions'` (anglais) |
| `campaigns.js`, `forms.js`, `evaluations.js` | `'Réservé aux admins et RH'` (français) |
| `events.js`, `resources.js` | `'Forbidden'` (anglais court) |

**Impact** : le client doit gérer 3 chaînes différentes pour le même type d'erreur.

---

### 2. Langue incohérente dans les messages d'erreur

| Anglais | Français | Endpoints concernés |
|---|---|---|
| `'User not found'` | `'Utilisateur introuvable'` | `users.js` vs `offboarding.js` |
| `'Email already exists'` | `'Email invalide'` | `users.js` (POST) |
| `'Insufficient permissions'` | `'Réservé à l'administrateur'` | Mélangé partout |

Aucune convention de langue uniforme n'est appliquée.

---

### 3. Pagination optionnelle incohérente (dual-mode)

4 endpoints utilisent un comportement **dual** : array brut sans `?page`, ou `{ data, total, page, limit }` avec `?page` :

- `GET /api/users` — array (100 max) / paginé
- `GET /api/campaigns` — array (100 max) / paginé
- `GET /api/forms` — array (100 max) / paginé
- `GET /api/evaluations` — array (200 max, marqué "Legacy") / paginé

Endpoints **jamais paginés** (array brut uniquement) :
- `GET /api/events` — 100 max (hardcodé)
- `GET /api/resources` — 200 max (hardcodé)

Endpoint avec **pseudo-pagination** (objet `{data}` sans métadonnées) :
- `GET /api/evaluations/history` — retourne `{ data: [...] }` sans `total`, `page`, `limit`

Endpoints **toujours paginés** (bon) :
- `GET /api/admin/audit`
- `GET /api/offboarding`

### 4. Codes HTTP incorrects ou surprenants

| Endpoint | Code actuel | Code attendu | Raison |
|---|---|---|---|
| `POST /api/evaluations/bulk` (succès partiel) | **207** Multi-Status | 200 ou 201 + corps explicite | 207 n'est pas standard pour une API REST JSON. Le corps `{ created, skipped, message }` suffit avec un 200. |
| `DELETE /api/forms/:id` | **204** No Content | 200 `{ deleted: true }` | Seul endpoint DELETE sans corps. Les autres (`events`, `resources`, `offboarding`, `campaigns`) retournent tous 200 avec un corps. |
| `GET /api/users/:id/gdpr-export` | **200** + header `Content-Disposition: attachment` | Comportement correct mais inattendu via un GET | Un export RGPD sensible sur un GET sans corps peut surprendre — attendu par certains comme un POST. |

### 5. Auth guard manquant ou surprenant

**`/api/evaluations/bulk` monté sans `authenticated`** :
```js
// index.js ligne 139
app.use('/api/evaluations/bulk', mutationLimiter)          // ← pas de authenticated ici
app.use('/api/evaluations', authenticated, evaluationRoutes) // ← authenticated ici
```
En pratique, `authenticated` s'applique via le mount `/api/evaluations` (préfixe parent), donc la sécurité n'est **pas compromise** — mais l'intention du code est trompeuse. Un lecteur pourrait penser que `/bulk` est public.

**`GET /api/evaluations/export` — aucun authGuard au niveau du montage** :
Le contrôle `ADMIN_ROLES.includes(req.user.role)` est fait dans le handler. `req.user` est garanti par `authenticated` au montage — cohérent mais entièrement dépendant des handlers.

---

### 6. Rôle `director` toujours présent dans le code (legacy non nettoyé)

Le rôle `director` est censé avoir été supprimé (remplacé par `manager`) mais subsiste dans :

| Fichier | Ligne | Usage |
|---|---|---|
| `index.js` | 127 | `authenticated = authGuard([..., 'director', ...])` |
| `users.js` | 18 | `!['admin', 'hr', 'director'].includes(req.user.role)` |
| `campaigns.js` | 32 | `ADMIN_ROLES_LOCAL = ['admin', 'hr', 'director', 'manager']` |
| `evaluations.js` | 86 | `role === 'manager' \|\| role === 'director'` |
| `evaluations.js` | 267 | `if (role === 'director')` — bloc de contrôle d'accès dédié |
| `evaluations.js` | 560 | `newEvaluator.role` doit être `'manager'` **ou `'director'`** pour une réaffectation |
| `evaluations.js` | 644, 662, 674, 684 | Permissions score/comment/objectives conditionnelles sur `'director'` |

**Impact** : les comptes legacy `director` obtiennent des accès larges (équivalent manager) dans plusieurs handlers, ce qui peut causer des comportements inattendus.

---

### 7. Nommage incohérent dans les réponses

**`GET /api/auth/me`** retourne à la fois `id` (string) et `_id` (ObjectId) :
```js
res.json({ id: user._id, ...user })   // user contient déjà _id → duplication
```
La réponse contient `{ id: "...", _id: "...", ... }` avec deux clés pour la même valeur.

**`GET /api/campaigns`** retourne des campagnes avec `createdBy` peuplé (`{ firstName, lastName, email }`), mais `POST /api/campaigns` retourne uniquement `{ id }`. Asymétrie : la création ne renvoie pas la ressource complète.

**`POST /api/forms`** et **`PATCH /api/forms/:id`** retournent `{ id }` uniquement, alors que tous les autres PATCH retournent la ressource complète (users, campaigns, evaluations…).

### 8. Features incomplètes ou partiellement implémentées

**`/api/analytics` — endpoint unique sans données JSON** :
- Seulement `GET /api/analytics/export/pdf` — un PDF all-campagnes sans filtre
- Aucun endpoint JSON analytics global (les données JSON par campagne sont sur `GET /api/campaigns/:id/analytics`)
- Aucun filtre (période, campagne, département) sur l'export PDF → données toujours globales

**`PATCH /api/events/:id` — champs non modifiables après création** :
- `campaignId` et `targetRoles` ne peuvent pas être mis à jour via PATCH
- Le handler n'inclut que `{ title, date, type }` dans les champs modifiables
- Aucun message d'erreur si le client envoie `campaignId` dans le body — silencieusement ignoré

**`PATCH /api/evaluations/bulk` — action `'archive'` ambiguë** :
```js
const allowed = roleTransitions[ev.status] || []
if (allowed.length > 0) {
  ev.status = allowed[0]   // prend le PREMIER statut valide — pas nécessairement 'archived'
}
```
L'action `'archive'` ne garantit pas que le statut final est `'archived'`. Elle applique la première transition disponible selon le rôle.

**`POST /api/users` — tempPassword non haché** :
```js
const tempPassword = require('crypto').randomBytes(16).toString('hex')
// ...
passwordHash: tempPassword,   // ← stocké sans bcrypt
```
La valeur stockée dans `passwordHash` est un hex brut, pas un hash bcrypt. Si `bcrypt.compare()` est utilisé à la connexion (ce qui est le cas dans `/api/auth/login`), ce compte ne pourra **jamais** se connecter avec ce mot de passe temporaire.

**`GET /api/evaluations/export` — filtre `dept` hors MongoDB** :
Le filtre par département est appliqué côté JavaScript après avoir fetché jusqu'à 5 000 enregistrements depuis MongoDB. Le filtre `dept` n'est pas passé à la requête Mongo.

---

### 9. Rate limiting asymétrique sur les évaluations

`PATCH /api/evaluations/:id` et `POST /api/evaluations` (création unitaire) ne sont soumis qu'au `apiLimiter` global (2 000 req/min), **sans** `mutationLimiter` (500 req/min).

En revanche, les routes bulk, users, campaigns, forms, events, resources ont toutes le `mutationLimiter`. L'oubli concerne les mutations unitaires sur `/api/evaluations`.
